## ADR-012: Hexagonal Validation Boundary (Pydantic at Ports, Dataclasses Internally)

**Status:** Accepted

**Context:** The system has two distinct categories of data structures: objects that cross a trust boundary (LLM response parsing, FastAPI request bodies, JSON persistence) and objects that flow between internal components after validation has already occurred. Using Pydantic for both conflates schema validation with domain modeling and adds per-instantiation validation overhead on every internal object construction.

**Decision:** Pydantic BaseModels are used exclusively at system ports where untrusted or untyped data enters or exits: FastAPI request/response bodies, LLM response parsing, and persistence boundaries requiring JSON roundtrip. Internal domain entities use `@dataclass(frozen=True)`. `AuditEvent` is Pydantic because it crosses the persistence boundary on every write and must serialize reliably to JSON. All entities in `core/entities.py` and `core/results.py` are frozen dataclasses because they flow only between already-validated internal components.

**Rationale:** Validation at the boundary is sufficient and correct. Re-validating on every internal object construction is overhead without safety benefit: by the time data reaches internal domain entities, it has been validated at the port. Frozen dataclasses are measurably faster in the hot path and make immutability explicit rather than relying on Pydantic's optional `frozen=True` config. The boundary distinction is also architecturally visible: a reader seeing `BaseModel` immediately knows they are looking at a port-facing object; a reader seeing `@dataclass(frozen=True)` knows they are looking at an internal domain entity.

**Rejected Alternatives:** Pydantic throughout. Rejected because it obscures the boundary distinction and adds unnecessary validation overhead on internal objects. Plain dataclasses (mutable) for internal entities. Rejected because mutability on domain entities creates hidden state change risks; `frozen=True` enforces the invariant that domain entities are value objects.

**Consequences:** This pattern is a permanent architectural constraint. Any new type added to `core/` must be classified as port-facing (Pydantic) or internal (frozen dataclass) before implementation. The classification is documented in the module docstring. Consistent with the Invoice Agent pattern established in prior projects.
