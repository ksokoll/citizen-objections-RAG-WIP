# ADR-030: Durable Append, Single-Writer Lock, and Recovery for the Audit Store

Status: Superseded by the Round 21 rollback (deliberate scope decision; see the
banner below). Section 2 stays in force.
Date: 2026-06-13
Deciders: Kevin Sokoll

> Round 21 rollback (2026-06-17). The three mechanisms this ADR decided, fsync
> durability (section 1), the single-writer advisory lock (section 3), and
> quarantine recovery with the WIEDERHERSTELLUNG heal event (section 4), were
> deliberately rolled back as out of demo scope. An ownership review found them
> infra depth that overweights the chain for the demo and is target-far for an
> ML/AI portfolio (the applicant builds ML/AI, not systems/infra). The concept
> this ADR set out to demonstrate, a manipulation-evident log, does not depend on
> them: canonical serialization, the hash chain with genesis and sequence
> numbers, verify_chain, the Form-B content-free payload gate, head anchoring,
> and fail-closed in _emit all survive the rollback and stay. Fail-closed in
> particular survives because the append still raises on OSError; it just no
> longer carries the fsync promise.
>
> What the rollback changes, recorded honestly rather than quietly dropped:
> - No crash-durability guarantee. The append writes and flushes but does not
>   fsync, so a power loss between the flush and the OS writeback can lose the
>   last appended line. The head still advances only after a successful write, so
>   it never claims a line the write call did not accept.
> - No concurrency guard. With the lock gone the chain is not guarded against a
>   concurrent writer. The demo has one synchronous writer (15.2), so the lock
>   was functionless; this is now stated, not assumed away.
> - No automatic tail recovery. A damaged tail (an unparseable line, or a last
>   line whose hash does not chain) now makes the store raise AuditLogError at
>   open instead of quarantining the bytes and continuing. "If the chain is
>   damaged, I do not open" is less code and a clearer demo statement. The
>   WIEDERHERSTELLUNG AuditEventType, its payload schema, and its AUDIT_RECOVERED
>   log event are removed with it.
>
> Deferral record (built, then deliberately rolled back for demo scope;
> production would restore them):
> - fsync durability (a crash-durable append before the head advances).
> - the single-writer advisory lock (a concurrent-writer guard).
> - quarantine recovery with the WIEDERHERSTELLUNG heal event (automatic tail
>   repair instead of a loud failure at open).
>
> This is a stronger deferral than "not built": it shows command of the topic
> plus the scope discipline to remove what the demo does not need. ADR-027's
> threat-model deferrals (HMAC, WORM, RFC 3161) are unchanged.
>
> Sections 1, 3, and 4 below are superseded by this rollback. Section 2 (the
> in-memory head as the sole duplicate mechanism, replacing the O(n) scan) stays
> in force: the head is still the duplicate mechanism and publish still does not
> scan the file. Read the rest of this ADR as the record of what was built and
> why, which the rollback then judged out of scope.

## Context

ADR-024 made the audit trail tamper-evident by hash-chaining each AuditEvent,
and ADR-029 pinned the canonical serialization the chain hashes over. ADR-029
left three mechanisms explicitly for this phase: the in-memory head as the sole
duplicate mechanism (replacing the O(n) per-publish scan), a durable append, and
a single-writer lock. ADR-027 already depends on those invariants by name: its
fail-closed policy (Round 19) is only diagnosable if an aborted run never leaves
the in-memory head ahead of disk, if a degrading store cannot interleave with a
second writer, and if a restart can recover the head rather than orphan events.

This ADR records the decisions that build those invariants. They are the kind of
correctness no test fully pins, because the failure is a crash at a specific
instant or a second writer at a specific instant: durability ordering (what is on
disk when the process dies between two statements), lock scope (which operations
must be mutually exclusive against another start), and recovery semantics (what a
restart does with a half-written tail). A subtle ordering bug here is invisible
until a crash months later produces a head that claims an event the disk lacks.

## Decision

### 1. Durable append: fsync before the head advances

The append sequence is: write the line, flush the Python buffer, os.fsync the
file descriptor, and only then advance the in-memory head (last hash, last
sequence). The head is the store's claim that an event exists; advancing it
strictly after the fsync means the claim never precedes its evidence on disk. A
crash between the fsync and the head-advance leaves a durable event the head has
not yet claimed (recovery re-seeds from it, no loss); a crash before the fsync
leaves no event the head asserts. The forbidden state, a head asserting an event
the disk lacks, cannot occur.

This preserves ADR-024's "advance only after a successful write" and tightens it:
the fsync is now inside the success condition. Any failure before the head-advance
(write or fsync) raises AuditLogError and leaves the head where it was, so a
failed append leaves the in-memory chain level with disk, never ahead of it.

The OSError boundary contract (ADR-027) holds: a write or fsync OSError is wrapped
as AuditLogError with the original chained, so callers route the recoverable class
without depending on stdlib exception types.

### 2. The in-memory head is the sole duplicate mechanism

publish no longer reads the file. The O(n) full-file duplicate scan ADR-027
accepted as interim (S6, the quadratic append) is removed: a run of n events was
O(n^2), and the in-memory head replaces it. The file is read exactly once, at
open, by the recovery path that seeds the head (section 4), so seeding is never a
second reader divergent from recovery.

A consequence, accepted deliberately: event_id duplicate detection is dropped.
The head tracks the last hash and sequence, not the set of all event_ids; keeping
duplicate detection would reintroduce an O(n) read or an O(n) in-memory set, both
of which contradict "the head is the sole mechanism." The pipeline keys each event
with a fresh UUID, so a re-published id is not a real scenario. This is a trade,
not a gap: the chain still records every append, and a forged or repeated id
produces a distinct chained event rather than a silent overwrite.

### 3. Single-writer advisory lock, re-entered per critical section

An advisory filelock (cross-platform, works on the Windows host) covers
open+recovery and each append. Two starts therefore cannot recover the same file
concurrently, and two writers cannot interleave an append. Acquisition blocks up
to a finite timeout and then fails loud (filelock.Timeout translated to
AuditLogError), so a contended store surfaces the contention on the documented
type rather than hanging or leaking a third-party exception.

The lock is re-entered per critical section (open+recovery in __init__, each
publish), not held for the store's lifetime. The reason is structural: two store
instances on one path are a supported pattern (reopen to continue the chain, as
the chain-continues-after-reopen test exercises), and a lifetime hold would
deadlock the second instance, because the cross-instance OS lock blocks even
within one process on Windows. Per-section acquisition lets sequential instances
hand the lock off while still making any concurrent writer fail loud.

The residual this leaves is named, not hidden: a writer slipping between two of
this store's locked sections would leave the in-memory head stale (the head is
only re-seeded at open). That is the accidental-concurrency limit the lock
accepts, not an interleaved-append race within a section. It is bounded by the
same threat model ADR-027 already declared: the lock guards against accidental
concurrency (a retry thread, a cron replay), not a deliberate actor with write
access who appends outside the store. The single-process assumption (Round 15.2)
holds on the deployment host: no second process holds the file.

### 4. Recovery: quarantine the damaged tail, record a recovery event

At open, the store reads the chain to its last valid line and seeds the head from
it. A damaged or partial last line is not truncated silently. It is one of two
EOF cases: a last line that fails to parse (a crash mid-write), or a last line
that parses but whose hash does not chain from its predecessor. In either case
the damaged line is moved to a quarantine file (audit.jsonl.corrupt.<timestamp>),
the live file is rewritten without it (durably, fsynced), and a recovery event is
appended to the chain.

Quarantine, not truncate: the damaged bytes are kept for later inspection rather
than discarded, and the SHA-256 of those bytes plus a line count are recorded so
what was removed is attributable. The raw quarantined content never enters the
chain or the logs.

The recovery action is itself a chain event. It is recorded two ways, matching
the system's existing custody-versus-observability split:

- A custody record in the chain: a WIEDERHERSTELLUNG AuditEvent
  (AuditEventType, core/events.py) carrying the quarantined bytes' hash and a
  line count in its payload, with a fixed non-objection sentinel einwendungs_id
  ("SYSTEM"). It chains from the last valid event, so the chain is continuous
  across the heal and a later append chains from the recovery event.
- An operational log event: AUDIT_RECOVERED, declared in the audit_log
  per-context vocabulary (events.py) and emitted at WARNING when recovery runs.
  Its fields (quarantined_hash, quarantined_lines) are declared in the audit_log
  key set, so it is surfaced through the 17.1 root-assembled union and allowlist
  like every other context event. The log line is alertable where the chain is
  not surfaced operationally until verify.

Only the last line is healed. A mid-file (interior) break is not healed here: an
interior line that fails to parse raises rather than silently truncating the tail
after it. The two cases are genuinely different. A partial last line is directly
observable: it is the realistic crash-mid-write outcome, and the head and the
file disagree by exactly one line. A lost whole event in the interior is not
observable from the file alone, because a rewriter can recompute the chain from
the break onward (ADR-027 threat model, rewrite-from-N) and a truncation leaves a
shorter chain that still verifies. Surfacing interior breaks needs verify_chain
and the external anchor (18c), not open-time recovery, so recovery deliberately
does not reach for it.

## Rationale

- The durability ordering is the whole point of the phase: the head is a claim,
  and a claim must not outlive its evidence across a crash. fsync-before-advance
  is the minimal ordering that guarantees it, and it composes with ADR-027's
  fail-closed abort (an aborted run never leaves the head ahead of disk).
- Removing the O(n) scan is the cost half of ADR-029's deferral; the head was
  always going to be the mechanism, and keeping a second duplicate check (a set
  or a re-read) would have been the duplicated-knowledge anti-pattern.
- The lock guards the accident the deployment can actually produce (a retry, a
  replay) and fails loud on it, while honestly declaring the deliberate-actor and
  cross-section residuals rather than overclaiming exclusion.
- Quarantine-not-truncate keeps the system from being the thing that destroys
  evidence: a damaged tail might be the only trace of what went wrong, so it is
  preserved and hashed, and the heal is itself recorded in the tamper-evident
  chain.

## Alternatives Considered

1. Advance the head before fsync (or without fsync). Rejected: a crash then
   leaves the head claiming an event the disk lacks, the exact forbidden state;
   it would also make ADR-027's fail-closed abort undiagnosable.
2. Hold the lock for the store's lifetime. Rejected: it deadlocks the supported
   reopen pattern (two instances on one path), because the OS lock blocks
   cross-instance even within one process on the Windows host. Per-section
   acquisition keeps single-writer exclusion while allowing sequential handoff.
3. Keep event_id duplicate detection via an in-memory set. Rejected: it
   reintroduces O(n) memory and contradicts "the head is the sole mechanism";
   the pipeline's fresh-UUID keying makes the scenario unreal.
4. Truncate the damaged tail silently. Rejected: it makes the system destroy the
   one trace of the failure and leaves no record that a heal happened, which is
   indistinguishable from tampering after the fact.
5. Heal interior breaks at open too (seed from the longest valid prefix, drop the
   rest). Rejected: it silently discards every event after the break, which is
   exactly the truncation a tamper-evident trail must not perform itself;
   interior breaks are for verify_chain and the external anchor to surface (18c).
6. A third-party single-writer abstraction or a DB. Rejected as disproportionate
   for a single-host, single-process pipeline; an advisory file lock is the
   smallest mechanism that meets the accidental-concurrency threat model.

## Consequences

Positive:

- A crash cannot leave the head asserting an event the disk lacks; durability
  ordering is explicit and tested via a seam on fsync.
- A run of n events is O(n) appends, not O(n^2); the interim quadratic append
  (ADR-027, S6) is closed.
- A concurrent writer fails loud instead of interleaving; the chain stays valid.
- A crash-mid-write tail is recovered automatically, the damaged bytes are
  preserved and attributable, and the heal is itself a chain event.

Negative:

- event_id duplicate detection is gone; publish is no longer idempotent on
  event_id. Accepted (the pipeline keys fresh ids).
- The lock is per-section, so a writer between two locked sections would leave
  the head stale. Accepted as the accidental-concurrency limit (ADR-027 threat
  model); a deliberate second writer is out of scope.
- Recovery rewrites the live file (the one operation that does), so the file is
  briefly not append-only at open. Bounded to the single quarantine heal under
  the lock, and itself fsynced.

Neutral:

- WIEDERHERSTELLUNG is a new AuditEventType member. It is a store-integrity
  custody event, not a pipeline stage, and lives in core because
  AuditEvent.event_type is typed against the enum.
- Interior-break detection, verify_chain as a named function, the payload
  allowlist, and head anchoring remain 18c. The fail-closed raise and the
  read-access audit event remain Round 19.
