# ADR-033: Fail-Closed Armed, End-to-End Ordering, and the Read Path Under Custody

Status: Accepted
Date: 2026-06-15
Deciders: Kevin Sokoll

> Round 21 rollback note (2026-06-17). Fail-closed (Decision 1), the read-access
> chain event (Decision 3), and the completion-before-return ordering (Decision
> 2) all stay. What changed underneath: ADR-030's fsync durability was rolled
> back as out of demo scope (ADR-030 superseded), so read "durable append (fsync
> before head-advance)" here as "written and flushed before return". The
> guarantee this ADR rests on is unchanged in substance: the completion event is
> recorded before the briefing is returned, and a completion-write failure aborts
> the run, so no output is handed back ahead of its recorded completion proof.
> Only the crash-durability promise (bytes on stable storage) is dropped.
> recover()/verify_open() still gate show-document; recover() now raises on a
> damaged tail instead of healing it.

## Context

ADR-027 ratified fail-closed for the six custody events but left the mechanism
as later work: the interim `Pipeline._emit` logged a governed
AUDIT_APPEND_FAILED ERROR and the audit_write_failures_total metric, then
swallowed the failure and returned the briefing. The abort was deferred until
the chain was robust enough that an abort signals a real problem rather than a
self-DoS (Sec-3) or a store quirk. ADR-024/029/030/031 built the durable,
recoverable, tamper-evident chain; ADR-032 made it content-free by declaration,
split read/write validation (closing the self-DoS), and made the common open
O(K). That prerequisite is now met: a custody-write failure is a real problem,
so the gate is armed. First the ledger reliable, then the gate.

ADR-027 also named, in its "Raw-store read path" section, a chain-level
read-access audit event as Round C work: the one read path on raw PII
(show-document) left only a deletable, retention-bound log trace (H4/S4), so
custody was asymmetric. The write paths were tamper-evident; the read path was
not. This ADR closes both: it arms the abort and makes custody symmetric.

Two points from ADR-027 are settled differently here, recorded so the divergence
is deliberate, not drift:

- ADR-027 sketched a new AuditWriteError to carry the abort. This ADR uses the
  existing AuditLogError. See Decision 1.
- ADR-032 already noted (its Consequences, "Neutral") that the fail-closed raise
  and the read-access event remained Round 19. This is that round.

## Decision

### 1. Fail-closed armed in the emit path (all six custody events)

`Pipeline._emit` re-raises the AuditLogError after recording it (the
AUDIT_APPEND_FAILED ERROR log and the audit_write_failures_total increment, the
visibility built in Round 15.x as the interim half of this gate). The error
propagates out of `run()`, so the run aborts and no briefing is returned that
would implicitly claim a complete custody trail it does not have. The abort
applies uniformly to all six custody events (EINGANG, TRIAGE, RETRIEVAL,
BRIEFING_ERSTELLT, KEIN_TREFFER, PIPELINE_FEHLER): a missing TRIAGE event is as
much a hole as a missing completion event, so the trail is lossless or it is not.

The abort is carried by the existing AuditLogError, not a new type. ADR-027
proposed AuditWriteError so callers could distinguish a write failure from a
duplicate or a query error. ADR-030 removed duplicate detection (the store keys
the chain off an in-memory head and does not scan per append), so there is no
duplicate error to distinguish from; and the one caller, the pipeline emit,
routes the whole recoverable class identically. A second exception type would be
ceremony with no consumer. A custody-write failure is a custody-write failure
wherever it occurs, so one type carries it.

Only the recoverable store-failure class is routed this way. Programming errors
(TypeError, ValueError, a publish-path bug) already propagated after the 18d
narrowing; this adds the recoverable class to what propagates. The publisher
contract obliges every store to translate raw I/O failures into AuditLogError, so
a raw OSError arriving at the emit is a contract violation that propagates like a
programming error.

The guarded `_log.error` stays: a sabotaged or degraded sink degrades visibility
only, it never masks or replaces the AuditLogError that aborts the run. The
sink-independent metric is incremented before the log attempt, so the failure is
counted even when the sink is also down.

### 2. End-to-end ordering: completion durable before return

The completion custody event (BRIEFING_ERSTELLT or KEIN_TREFFER) is emitted,
written, and flushed before `run()` returns the briefing (ADR-030's fsync was
rolled back in Round 21, so the append is no longer crash-durable; the ordering
and the abort-on-failure are unchanged). The return value is the system's claim
that the objection was processed and recorded; that claim must not precede its
record on disk.

Why the window matters: without this ordering there is a window in which an
output exists, a briefing handed to the Sachbearbeiter, whose completion proof
the disk does not yet hold. A crash in that window leaves a delivered briefing
with no custody record that it was produced. Emitting the completion event
before the return, combined with fail-closed (Decision 1), closes the window:
the emit completes before the return statement, and if the write fails the run
aborts and no briefing is returned. There is no path on which output exists
ahead of its recorded completion proof.

### 3. Read-access chain event (custody covers the read path)

The show-document path records a ROHDOKUMENT_ZUGRIFF event in the chain when it
reads a stored raw document (unmasked PII) back out, before the content is
printed, synchronously and fail-closed. The document is loaded first (which
validates the id and emits the operational raw_document_accessed log event), then
the read-access event is written, and the content is printed only if that write
is durable. A failure anywhere in the audit path, lock contention, a tampered
tail caught at verify_open, or the append itself, aborts the read with a nonzero
exit and a clear refusal, and prints nothing. Raw-PII access happens only when it
is provably recorded in the immutable chain.

show-document opens the audit store as a writing path (recover then verify_open,
like process): recover seeds the chain head so the read event chains onto the
real tail rather than re-seeding genesis, and verify_open refuses to disclose PII
onto a chain whose tail does not verify.

Sentinel choice: the event's einwendungs_id is the document_id being read, not
the SYSTEM sentinel. WIEDERHERSTELLUNG and STARTKONFIGURATION are process-wide,
tied to no objection, so they carry the SYSTEM sentinel. A raw-document read is
not process-wide: it is access to one specific objection's PII, so the natural
correlation is that objection's id. Carrying it means a query for everything
touching one objection surfaces its read accesses beside its pipeline events,
which a SYSTEM-keyed event would not.

Content-free payload (Form B, ADR-032): the event declares document_id (str) and
nothing else. The document_id is redundant with the einwendungs_id by
construction (both set from the one id), the same self-describing duplication
EINGANG carries, and it can never carry the document text. The "when" is the
event's own top-level timestamp, already inside the canonical hash and so
tamper-evident; it is not duplicated into the payload, which would be a second
clock that could diverge from the hashed one without adding anything the hash
does not already protect.

The operational raw_document_accessed log event stays (ADR-027): the log and the
chain event are a pair from the same access, like the recovery event's log+chain
pair. The log is the alertable operational trace, retention-bound; the chain
event is the undeletable custody record that survives retention.

### 4. ADR-027 interim notes resolved on the write side

ADR-027's "Interim risk: the double failure" named the store-down-and-sink-down
case as an accepted interim risk, to be closed from both directions: Round B's
sink-independent metric, then Round C's fail-closed raise. Round B landed the
audit_write_failures_total metric. This round lands the raise. The interim is
over on the write side: a persistent custody-write failure now aborts the run
rather than returning unproven output, even when the log sink is also down,
because the AuditLogError propagates regardless of whether the ERROR line could
be written.

Residual, stated rather than assumed away: a simultaneous store-and-sink failure
still degrades VISIBILITY. If the store is down and the sink cannot persist the
ERROR line, the only durable signals are the sink-independent metric and the
abort itself; the located ERROR record is lost for that event. What no longer
degrades is CORRECTNESS: the run aborts, so no unaudited briefing is produced.
The remaining exposure is purely observational, not a completeness hole.

## Threat model

Unchanged from ADR-027 and ADR-032. Fail-closed plus the read-access event keep
the chain complete (no silent missing write) and now symmetric (the read path is
recorded), but the chain remains tamper-evident, not tamper-proof, and
content-free by construction. The deferrals that close the residual gaps are
unchanged, each with its stated trigger: HMAC with a managed key
(rewrite-from-N), WORM storage (tail truncation and in-place rewrite at the
storage layer), RFC 3161 qualified timestamps (the NTP assumption), and SIEM
export (central monitoring). This round adds no chain mechanics. The head into
the publisher protocol (A4 first part) and the dedicated anchor command remain
backlog.

## Availability trade-off

Fail-closed trades availability for completeness, the trade ADR-027 accepted. A
transient store failure now aborts that objection's processing; the raw document
is already persisted at ingestion, so nothing is lost and a re-run recovers once
the store is healthy. For show-document a chain failure refuses disclosure rather
than handing out PII without a record; the operator fixes the store and re-reads.
Availability loss is bounded and the accepted cost of a complete, symmetric trail.

## Alternatives Considered

1. Keep a new AuditWriteError type (the ADR-027 sketch). Rejected: no consumer
   distinguishes it from AuditLogError. Duplicate detection is gone (ADR-030), so
   there is no duplicate failure to tell apart, and the single caller routes the
   whole recoverable class identically. The type would be ceremony without a
   reader.
2. Write the read-access event to the log only, or after printing the content.
   Rejected: a log is retention-bound and deletable, the exact H4/S4 gap; and
   disclosing before the durable custody record reintroduces output-ahead-of-
   evidence on the read side. The chain event must precede disclosure.
3. Carry the SYSTEM sentinel on the read event, uniform with startup/recovery.
   Rejected: it would make a per-objection access indistinguishable from
   process-wide events and break the "everything touching one objection" query.
   The access is objection-specific; its correlation should be too.
4. Duplicate the access timestamp into the read event's payload. Rejected: the
   AuditEvent already carries a top-level timestamp inside the canonical hash. A
   second payload timestamp is a redundant clock that can diverge from the hashed
   one and adds nothing the hash does not already protect.
5. Leave show-document a pure read with no chain write (status quo). Rejected: it
   leaves custody asymmetric, the defect this ADR closes.
6. Fail-open with a non-auditable marker on the briefing (ADR-027 alt 2). Still
   rejected for the same reason: it pushes a compliance decision onto the
   Sachbearbeiter at read time and defines a second class of output.

## Consequences

Positive:

- The system cannot produce a briefing whose custody trail the chain lacks, and
  the read path cannot disclose raw PII without an undeletable record. Custody is
  complete and now symmetric across write and read.
- There is no window where an output exists ahead of its durable completion
  proof on disk.
- Audit-store degradation surfaces immediately (a failed run or refused read, the
  metric, the ERROR log) instead of at the next verify_chain().
- The double-failure interim risk from ADR-027 is closed on the write side: a
  store-and-sink double failure degrades visibility only, never correctness.

Negative:

- A transient store failure aborts that objection's processing or refuses a
  show-document read; operator intervention or a re-run is required. The accepted
  availability cost (ADR-027).
- show-document is now a writing path: it takes an --audit-log path and runs
  recover()/verify_open(), so a tampered tail refuses the read.

Neutral:

- The custody event set must stay in sync if AuditEventType grows; the payload
  schema's "every event type declares a schema" test (ADR-032) fails closed on a
  new member that is not classified.
- BOUNDED_CONTEXTS.md's error-propagation table and Coordinator map are updated
  in this round to reflect the fail-closed posture (the edits ADR-027 deferred to
  the Round C implementation), so the architecture doc no longer encodes the old
  fail-open behavior.
