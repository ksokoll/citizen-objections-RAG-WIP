# ADR-022: Deterministic Briefing Instead of LLM-Generated Würdigung

## Status

Accepted.

## Context

ResponseDrafting is the final stage of the pipeline. The original design (bounded_contexts.md, before this decision) had it generate a Würdigung per argument via an LLM, grounded in the resolved Gesetzestext, with a §-reference verification gate setting UNTERDRUECKT_UNVERIFIED on any hallucinated reference.

Working through the generation step surfaced a structural gap. A Würdigung is an Abwägung: it weighs the citizen's argument, supported by a norm, against the concrete facts of the specific building project. The pipeline has the citizen's argument (from Triage) and the applicable norm with its source text (from Retrieval). It does not have the case-specific facts of the project: the Planunterlagen, Gutachten, textual Festsetzungen, Begründung, and Umweltbericht that make up the Akte. Those are outside the system boundary.

Without the case facts, an LLM could only restate the abstract legal position, not perform the concrete Abwägung. Generating a Würdigung anyway would either produce a generic non-assessment or invite the LLM to fabricate case details it does not have, which is precisely the hallucination surface the rest of the project has worked to eliminate (ADR-006 verification, the norm-resolution exact-match decision in ADR-021).

The project's stated focus is applying Behörde-appropriate observability. A deterministic, auditable output serves that focus better than an LLM assessment that would then need re-verification.

## Decision

ResponseDrafting produces a deterministic per-argument briefing, not an LLM-generated Würdigung. For each extracted argument it pairs the citizen's argument with its resolved norm text and presents a structured briefing entry for the Sachbearbeiter, who performs the actual Abwägung against the Akte.

No LLM is used in ResponseDrafting. There is no generation step and therefore no §-reference verification gate: nothing is fabricated, so there is nothing to verify. The missing case context is represented explicitly in the data model (the requires_case_context flag on each ready entry and the document-level limitation_note), not only in rendered prose.

## Rationale

This follows the project's context-management hierarchy: use the lightest mechanism that solves the problem and justify each escalation. LLM generation is an escalation. For the task ResponseDrafting can actually perform with the data available (pairing arguments with their applicable norms), a template is sufficient and an LLM adds cost, latency, non-determinism, and a hallucination surface with no offsetting benefit, because the data needed for a genuine Würdigung is absent regardless of model capability.

The decision is consistent with two earlier negative results in the project: the v3 tagged-context approach (rejected with data in Iteration 13) and the vector-fallback in norm resolution (rejected with data in ADR-021). In all three cases a more complex mechanism was measured or reasoned to add no value for the specific problem and was not adopted. Here the reasoning is structural rather than measured: the necessary input (the Akte) is outside the system, so generation cannot do the job it would be added for.

Determinism also serves the observability focus. Every briefing entry is traceable to a Triage argument and a Retrieval resolution, with no stochastic generation step in between. The output is fully auditable.

## Consequences

Positive:

- No hallucination surface in the final stage. The briefing contains only what was deterministically extracted and resolved.
- No §-reference verification gate needed, since no references are generated.
- Deterministic, reproducible, no LLM provider dependency, no token cost, no latency in ResponseDrafting.
- The scope limitation (no case-file-grounded assessment) is explicit in the data model and the rendered output, not hidden.
- Fully auditable: every briefing entry traces to its source argument and norm resolution.

Negative:

- The output is a structured briefing, not a finished Würdigung. The Sachbearbeiter still performs the formulation and the case-file Abwägung. The system reduces preparation effort but does not produce a draft response.
- The most ambitious version of the project (end-to-end objection to drafted assessment) is explicitly out of scope and deferred.

## Future Work

A case-file-grounded Würdigung is a coherent follow-up project. It would require a mock Akte (a realistic building-project dossier matching the test objections) and a second retrieval path: semantic vector retrieval over the unstructured case documents, distinct from the deterministic norm resolution. That is the natural home for vector RAG, which exact-match resolution displaced in this project (ADR-021). The retained experimental E5Embedder and FaissNormIndex (ADR-021 reversibility note) would apply there. This is deferred to a separate project where the Akte is part of the design from the start, rather than retrofitted here.

## Alternatives Considered

A. LLM-generated Würdigung grounded only in the norm text. Rejected. Without case facts the LLM can only restate the abstract legal position, which is not an Abwägung, or fabricate case details, which is the hallucination surface the project has eliminated elsewhere.

B. LLM generation with a mock Akte built now to supply case context. Rejected for this project's scope. Building a realistic mock Akte and a second semantic-retrieval pipeline is a substantial scope increase, effectively a second project. It deserves a design where the Akte is foundational, not appended to a pipeline conceived without it.

C. Deterministic briefing with the limitation made explicit. Chosen. It delivers the value the available data supports (argument-to-norm-to-text mapping), serves the observability focus through full auditability, and names the scope boundary transparently rather than papering over it with under-grounded generation.
