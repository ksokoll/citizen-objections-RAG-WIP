# ADR-027: Audit-Write Failure Policy: Fail-Closed for Custody Events

Status: Accepted (policy ratified in Round A; the fail-closed mechanism in Pipeline._emit is implemented in Round C, per OBSERVABILITY_IMPLEMENTATION.md)
Date: 2026-06-09
Deciders: Kevin Sokoll

## Context

Technical integrity of the audit trail (ADR-024) does not give completeness:
the chain proving that no recorded event was altered says nothing about events
that were never recorded. For a Behoerde compliance trail, a silently missing
entry is worse than a detectably broken chain: a broken chain fails
verify_chain(), a never-written event is invisible.

Current behavior is a completeness gap. Pipeline._emit catches every exception
from the audit publish, writes to stderr, and never re-raises. A failed publish
is swallowed: the briefing is returned and the pipeline reports success while
an event is silently absent. Combined with the in-memory chain-head planned in
ADR-024, a degrading audit store (disk full, lock contention, filesystem
latency) would produce months of successful briefings with missing links,
surfacing only at the next restart or verify_chain() run, with no alarm because
the swallow never reached a metric.

The trade-off is completeness versus availability: aborting on a failed audit
write means a citizen's objection is not processed during a transient storage
hiccup; continuing means the system can claim auditedness it does not have.

This policy is ratified before the code that depends on it is written
(pre-registration discipline applied to an architecture decision). Round A
makes the failure visible (the interim _emit logs a governed AUDIT_APPEND_FAILED
ERROR event instead of an ungoverned stderr print). Round C turns visibility
into the fail-closed abort specified here, once the chain invariants that make
an abort diagnosable exist (durable-append-before-head-advance, chain-head
recovery, single-writer lock; ADR-024).

## Decision

Fail-closed for the six custody events; best-effort is acceptable only for
operational telemetry outside the custody chain.

1. The custody events are EINGANG, TRIAGE, RETRIEVAL, BRIEFING_ERSTELLT,
   KEIN_TREFFER, and PIPELINE_FEHLER. For these, a failed durable append
   raises AuditWriteError (new exception in core/failures.py, distinct from
   AuditLogError so callers can distinguish a write failure from a duplicate
   or query error). The pipeline run aborts. No briefing is returned that
   implicitly claims to be audited when it is not.

2. End-to-end ordering. The completion event (BRIEFING_ERSTELLT or
   KEIN_TREFFER) must be durably appended before run() returns the briefing.
   The return value is the system's claim that the objection was processed and
   recorded; that claim must not precede its own evidence. A briefing handed to
   the Sachbearbeiter is therefore never ahead of the custody record that
   attests to it.

3. Failure during failure handling: if emitting PIPELINE_FEHLER itself fails,
   the AuditWriteError is raised with the original pipeline exception chained
   (raise ... from), so neither failure masks the other.

4. Known residual gap, accepted: if ingestion fails before a document_id
   exists, no event can be keyed and none is emitted. This is recorded here as
   a known boundary, not silently assumed. The system boundary (entry point)
   is the place to log such pre-identity failures operationally.

5. Visibility regardless of policy: every failed publish increments
   audit_write_failures_total and produces an ERROR status in the structured
   timing log. The metric exists so that even future best-effort paths cannot
   degrade invisibly.

6. Operational telemetry (structured logs, metrics, spans) is explicitly not
   custody data. A logging or metrics failure never aborts a run.

## Threat model

A policy that names a control objective must also name what it does not
protect against, or the overclaim is itself an audit finding. The keyless
hash chain (ADR-024) that this policy keeps complete protects against:

- Accidental corruption: a truncated, reordered, or partially written event
  breaks verify_chain() at that event and every successor.
- Naive single-event tampering: editing one event's content or timestamp in
  place breaks the chain at that event, because the timestamp and content are
  inside the hash and each event binds the predecessor hash.

It does not protect against:

- Rewrite-from-N: an actor with write access to the JSONL who recomputes every
  hash from the point of edit onward produces a chain that verifies clean. A
  keyless chain has no secret the rewriter lacks.
- Tail truncation: an actor who deletes the last k events and the corresponding
  head leaves a shorter chain that still verifies. Completeness against
  truncation is exactly what fail-closed plus the external head anchor address,
  and only partially: the external head anchor in the evaluation results.json is
  sparse (written at eval time, not per event) and is committed by the same
  actor, so it raises the cost of an undetected truncation rather than closing
  it.
- A cooperating-process bypass: the advisory single-writer lock (ADR-024)
  guards against accidental concurrency (a retry thread, a cron replay), not
  against an actor who deliberately appends to the JSONL outside the store.

What closes these gaps is out of scope here and recorded as deferrals below
(HMAC with a managed key defeats rewrite-from-N; WORM storage defeats
truncation; RFC 3161 timestamps defeat backdating beyond NTP trust). The
honest scope of this phase is integrity against accident and naive tampering
plus completeness against silent write failure, not insider-tamper resistance.

## Timestamp evidentiary value (NTP assumption)

Timestamps are inside the hash, so backdating an event after the fact breaks
the chain. The evidentiary value of the absolute timestamp, however, rests on
the host clock being correct at write time, which is assumed to be maintained
by NTP on the deployment host. This is an assumed property, not one the system
enforces: a host with a skewed clock writes a chain that is internally
consistent but absolutely wrong in time. Recorded here rather than silently
relied upon; qualified timestamping (RFC 3161) is the deferral that would
replace the assumption with evidence.

## Rationale

- For a public-administration trail, completeness is the point. The trail
  exists to prove afterwards that a given objection was processed in a given
  way. Output produced without its custody record undermines the purpose of
  the entire AuditLog context.
- Fail-closed converts an invisible failure mode into a visible, attributable
  one: the run fails loudly, the metric increments, the operator is forced to
  fix the store before further output is produced.
- The durable-append-before-head-advance invariant (ADR-024) guarantees that
  an aborted run never leaves the in-memory head ahead of disk, so fail-closed
  composes cleanly with chain recovery.
- Availability loss is bounded and acceptable at this volume: a transient
  storage failure delays processing; it does not lose the objection, because
  the raw document is already persisted at ingestion.

## Alternatives Considered

1. Fail-open (current behavior). Rejected: accepts silent incompleteness,
   which is the failure mode this layer exists to eliminate.
2. Fail-open with a non-auditable marker on the briefing. The pipeline
   continues but the result carries auditable=False. Rejected: it pushes a
   compliance decision onto the Sachbearbeiter at read time and creates a
   second class of output whose handling rules would themselves need
   definition. More machinery, weaker guarantee.
3. Queue-and-retry (buffer failed events, replay later). Rejected: a replay
   writer is a second writer, which ADR-024 forbids by design (single-writer
   lock); buffered events also break the timestamp-inside-hash ordering
   guarantee. Disproportionate for a single-process pipeline.
4. Fail-closed for all events including any future telemetry events.
   Rejected: couples operational noise to domain availability. The custody
   set is closed and named instead.

## Deferrals (recorded as decisions, not omissions)

Each of these closes a gap named in the threat model. None is built in this
iteration; each is a deliberate scope cut with a stated trigger.

- HMAC with a managed key over each event (defeats rewrite-from-N). Trigger:
  an insider-tamper threat that a keyless chain cannot address. Deferred
  because key management is its own infrastructure and the current threat model
  is accident plus naive tampering.
- WORM storage for the JSONL (defeats tail truncation and in-place rewrite at
  the storage layer). Trigger: a deployment target that offers append-only
  object storage. Deferred: out of scope for a single-host pipeline.
- RFC 3161 qualified timestamps (replaces the NTP assumption with evidence).
  Trigger: a legal requirement for trusted time. Deferred: requires a timestamp
  authority.
- SIEM export of the custody and ERROR events. Trigger: a central monitoring
  obligation. Deferred: the metrics and logs are local in this phase
  (OBSERVABILITY_IMPLEMENTATION.md, Out of scope).
- Periodic log-sink scan for non-allowlist keys. Re-justified as
  defense-in-depth, not the primary control: ADR-026 closes the main leak path
  at the sink by routing both structlog and foreign stdlib records through the
  same allowlist chain, so a periodic scan is now a backstop against a future
  misconfiguration rather than the load-bearing mechanism it would have been
  without stdlib routing. Deferred on that basis.

## Consequences

Positive:
- The system cannot produce output that falsely implies a complete trail.
- Audit-store degradation surfaces immediately (failed run, metric, ERROR
  log) instead of at the next verify_chain().
- The custody event set is now an explicit, named contract.
- The control objective is bounded by an explicit threat model, so the trail
  is not described as proving more than it does.

Negative:
- A transient storage failure aborts processing for that objection. Operator
  intervention or a re-run is required. This is the accepted cost.
- Pipeline behavior changes (in Round C): this is the one deliberate exception
  to the "observability layer does not change core pipeline behavior" scope
  rule, noted as such in OBSERVABILITY_IMPLEMENTATION.md.

Neutral:
- The custody set must be kept in sync if AuditEventType grows. A test
  asserting that every AuditEventType member is classified (custody or
  telemetry) guards the contract.
- BOUNDED_CONTEXTS.md error-propagation table requires an update (AuditLog
  write failure now aborts the run for custody events). The update lands with
  the Round C implementation, since until then the interim _emit still swallows
  after logging.
