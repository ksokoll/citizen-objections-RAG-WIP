# ADR-023: Observability as an Architecture Characteristic

Status: Accepted
Date: 2026-06-02
Deciders: Kevin Sokoll

## Context

The pipeline is functionally complete (DocumentIngestion -> Triage ->
Retrieval -> Briefing -> AuditLog) but has no operational layer. A
public-administration deployment needs structured logs, traces, and metrics,
all correlated per processed objection, so that any run can be reconstructed
and defended afterwards.

A prior project (an invoice-booking LLM agent) established a three-pillar
observability stack. The question is which parts transfer to this project,
given that the prior project was a stochastic agent with a coordinator loop,
variable iterations, and tool calls, while this project is a deterministic
linear pipeline with an LLM call only in Triage.

## Decision

Treat observability as an explicit Architecture Characteristic and adopt the
three-pillar model, retaining the full tool stack from the prior project but
re-specifying the instrument contents for a deterministic pipeline.

1. Structured logs via structlog. JSON for production, console for
   development, switched via OBSERVABILITY_FORMAT. Every event carries an
   ISO-8601 UTC timestamp, level, message, and correlation id.
2. Distributed tracing via OpenTelemetry. Flat, static span hierarchy:
   pipeline.run as root with one child span per bounded context. No
   iteration or tool-call levels, because the pipeline has none.
3. Metrics via prometheus_client. The tool is retained; the metric set is
   rewritten. Dropped (no equivalent in a deterministic pipeline):
   iterations-per-run, tool-calls, verification-failures. Optional:
   llm-tokens (Triage only). Kept: runs-total, run-duration. Added:
   objections processed (by type or status), processing duration,
   resolved-vs-unresolved norms (the quality signal), arguments per
   objection.
4. Correlation id via contextvars.ContextVar, anchored on the existing
   document_id rather than a separate uuid.
5. Span instrumentation primarily via a @traced decorator (Level 1), since
   the pipeline has static named steps.

PII discipline and corpus reproducibility are decided alongside this stack
because they are coupled to the logging path and the briefing data model
respectively; see the implementation plan. The tamper-evident audit trail is
a separate decision (ADR-024).

## Rationale

- Observability is listed as an Architecture Characteristic for ML systems
  in the engineering standards: you must be able to see what the system did,
  on what input, when. A pipeline without it fails that question.
- The three-pillar model is the industry standard and uses the vocabulary
  expected in regulated environments (public sector, banking, insurance),
  which makes the resulting skill transferable.
- structlog, OpenTelemetry, and prometheus_client are mainstream, stable, and
  backend-agnostic.
- Anchoring the correlation id on document_id avoids inventing a parallel
  identity: the document_id already is the natural per-objection key, and the
  trace, the logs, and the audit trail can all key on it.
- The span hierarchy is flat rather than nested because a deterministic
  pipeline has no iteration tree. Forcing the agent's nested structure here
  would add levels with nothing in them.
- Retaining the stack but rewriting the metrics keeps the learned tooling
  consistent while avoiding the anti-pattern of carrying over instruments
  (iterations, tool calls) that measure phenomena this system does not have.

## Alternatives Considered

| Dimension | No observability | Logs only | Three pillars (chosen) | Copy agent stack verbatim |
|---|---|---|---|---|
| Run reconstruction for audit | Impossible | Partial | Yes | Yes |
| Per-context latency | Impossible | Impossible | Yes (spans) | Yes |
| Metrics fit the pipeline | n/a | n/a | Yes (rewritten) | No (agent metrics) |
| Skill transfer | None | Limited | High | High |
| Effort | Zero | Low | Medium | Low but wrong fit |

"Logs only" was rejected because logs without spans cannot show where time
went. Copying the agent stack verbatim was rejected because the
agent-specific metrics (iterations, tool calls, verification failures) have
no referent in a deterministic pipeline and would be dead instruments.

## Consequences

Positive:
- Every objection run is fully traceable via the document_id-anchored
  correlation id.
- Per-context latency (notably the Triage LLM call and Retrieval) is visible.
- The resolved-vs-unresolved norm metric surfaces retrieval quality directly.
- The skill set transfers to any regulated Python service.

Negative:
- Three new dependencies (structlog, opentelemetry-sdk, prometheus-client),
  each small and stable.
- Instrumentation code is added to the pipeline; the @traced decorator keeps
  most of it off the business logic.

Neutral:
- No backend exporter in this phase: metrics live in the in-process registry,
  spans in the in-memory provider. Production would add an OTLP exporter and
  a Prometheus scrape endpoint, each a small addition.
- The OBSERVABILITY_FORMAT environment variable adds one configuration knob.
