# ADR-031: Chain Verification, Payload Allowlist, Anchoring, and Config-in-Chain

Status: Accepted; the payload allowlist (section 2) superseded by ADR-032
Date: 2026-06-15
Deciders: Kevin Sokoll

> Honesty note (Round 18d, ADR-032): section 2 below described the content-free
> guarantee as a payload allowlist whose string values were bounded to 128
> characters. That wording overclaimed. A length bound is not an erasure
> guarantee: `{"namen": ["Max Mustermann", ...]}` validated as content-free
> because every string fit the bound, so a name entered the chain unimpeded. The
> "honest residual" paragraph named this ("a 30-char string can still be a
> name") but then sold the mechanism as the content-free property anyway. The
> policy also lived in the shared kernel (core/events.py), binding every
> AuditEvent producer rather than the audit context that owns the chain. ADR-032
> replaces the length heuristic with a per-event key allowlist (Form B): each
> custody event declares exactly which payload keys and types it may carry, the
> store enforces that at write entry, and an undeclared key (a text fragment
> among them) is rejected by positive declaration, not by string length. Read
> sections 2 and the threat model below as superseded by ADR-032.

## Context

ADR-024 made the audit trail tamper-evident by hash-chaining each AuditEvent;
ADR-029 pinned the canonical serialization the chain hashes over; ADR-030 made
the append durable, the writer single, and a damaged tail recoverable. Three
things were named there as this phase and deferred: a verify path that uses the
exact same serializer as the write path (the silent-break risk ADR-029 exists to
prevent), the payload allowlist that keeps the chain content-free (the
precondition ADR-027's threat model already leans on by name), and the external
head anchor (named in ADR-027's threat model as the partial answer to
truncation). This ADR records those decisions, plus writing the startup_config
event into the chain so the active controls are provable after the fact.

The fail-closed raise in Pipeline._emit, the chain-based read-access audit event,
and a dedicated frequent anchor command remain out of scope (Round 19 / backlog),
recorded here as deferrals, not omissions.

## Decision

### 1. verify_chain: same serializer, first break with location, two call sites

A named verify_chain walks the chain and recomputes each event's hash via the
single canonical serializer from ADR-029 (compute_event_hash, which dispatches
canonical_bytes per serialization_version), not a second copy. This is the whole
point of having one serializer: a chain written under canonical_bytes verifies
only against that same byte form, so a second, divergent serialization
introduced into the verify path would make a freshly written chain fail to
verify. The same-serializer guard is therefore a test that verifies a freshly
written chain (and the 18a golden event): it fails exactly when the two
serializations drift.

verify_chain checks, per event, the monotonic sequence number, the chained hash
(content plus predecessor hash), and, on a full walk, the genesis sentinel. It
reports the first break as a ChainBreak (index, sequence number, reason,
expected vs found), not a bare boolean, so an auditor learns where and why.

Two call sites, by cost:

- A fast tail-window verification at store open verifies only the last K events
  (default 256), seeded from the event just before the window. ADR-030's
  recovery heals only a damaged last line, so a parseable-but-non-chaining
  interior line near the tail (a naive in-place edit) would otherwise pass open
  silently and surface only at the next audit. The tail window catches such a
  break at open and raises it loudly with location. A break before the window is
  the full walk's job; not doing it at open is the deliberate startup-speed cut.
- A CLI command, `python -m app verify-audit`, runs the full walk from genesis
  for an auditor and exits nonzero on a detected break with a located,
  content-free report. It is non-mutating by construction: it reads and parses
  the file directly (verify_chain_file) rather than opening the store, so an
  audit never triggers recovery or appends a recovery event, and an unparseable
  line is reported as a located break rather than crashing the command.

This is where an interior break, which ADR-030 deliberately left for this phase,
is diagnosed with detail rather than silently truncated.

### 2. Payload allowlist: default-deny, content-free by construction

A Pydantic validator on AuditEvent restricts payload values to int, float, bool,
strings up to a documented maximum length (128), or a flat (one-level) list or
dict of those. A value outside that set, a deeper nesting, or an over-length
string is rejected at construction, loudly, naming the offending key.

The chain stays content-free so it can coexist with the right to erasure: the
chain is undeletable, the right to erasure reaches only the raw store, and that
coexistence holds only if the chain carries nothing erasure-bound. Payloads
carry pseudonyms, hashes, and counts, never text fragments.

The flat-container leniency is a deliberate, recorded deviation from a
strict-scalar reading. The EINGANG event already carries a flat
masked_entity_counts map (dict[str, int]), and changing that payload is a
pipeline behavior change explicitly out of this round's scope. So the allowlist
admits one level of list or dict of simple leaves (counts maps, version maps),
which stays content-free by the same length bound and depth limit. A deeper
structure, which could nest arbitrary content, is rejected.

Honest residual, recorded at the constant and the validator and here: the
validator enforces types and length mechanically, but a 30-char string can still
be a name. This is mechanism plus policy plus a named residual, exactly like the
log-key allowlist (ADR-026), not a guarantee. The bound is a heuristic against
snippets, chosen to admit a SHA-256 hash (64 hex), short ids, version strings,
and entity labels while rejecting a pasted objection fragment.

### 3. Head anchoring: external witness, sparse and same-actor

An eval run records the chain head (hash plus sequence) into its committed
results.json (head_anchor produces a reserved chain_anchor block). Because
results.json is committed, the git history becomes an external trust anchor: a
keyless chain rewritten from a break onward still verifies internally, so an
external witness is what makes a truncation or rewrite detectable at all.

The anchor's limits are the point of recording it, not a flaw to hide. It is
sparse (written only at eval runs, so the detection window is the gap between
runs) and, in a one-person repo, committed by the same actor who could tamper.
It therefore raises the cost of an undetected truncation rather than closing it;
it demonstrates the concept of an external, distributed anchor. A dedicated,
more frequent anchor command is backlog, with the trigger a concurrent or
operational context where the eval cadence is too sparse to matter.

The store owns the head (the in-memory head ADR-030 maintains, so the anchor
never claims an event the disk lacks); the anchor shaping is the audit_log's
public form; writing results.json is eval glue under experiments/.

### 4. startup_config written into the chain

The startup_config event (a log event since Round 16) is, for the process
command, additionally written into the chain as a STARTKONFIGURATION event at
process start, using the same reserved system-id sentinel as the recovery event
(SYSTEM_EINWENDUNGS_ID): both are process-wide, not tied to an objection. It is
opened before any objection event, so it is the chain genesis: the controls are
attested before anything they govern is recorded.

Its payload is the content-free provenance the payload allowlist permits: git
sha, package versions (a flat version map), allowlist size, tracing flag, log
format, corpus hash, model id, admitted endpoint. The resolved store paths stay
in the log event only (S5): a path can exceed the payload length bound and is
not part of the proof. STARTKONFIGURATION is a new AuditEventType, a process-wide
custody event like WIEDERHERSTELLUNG; the sentinel is generalized into
core/events.py so both process-wide events share one non-objection id.

## Threat model update

The chain remains tamper-evident, not tamper-proof, and now content-free by
construction. Relative to ADR-027's threat model:

- Naive single-event tampering and accidental corruption are now not only
  breakable but diagnosable with location: verify_chain reports the first break,
  the tail window surfaces a tampered tail at open, and verify-audit walks the
  whole chain for an auditor.
- The payload allowlist makes the content-free property structural rather than
  conventional: a text fragment cannot enter the chain by accident, which is
  what licenses the chain coexisting with erasure (the chain is undeletable;
  erasure hits only the raw store).
- The external head anchor partially addresses tail truncation and
  rewrite-from-N (ADR-027): a keyless chain has no secret a rewriter lacks, so
  only an external witness can catch a consistent rewrite. The anchor is sparse
  and same-actor in this repo, so it raises the cost rather than closing the
  gap. HMAC with a managed key (rewrite-from-N), WORM storage (truncation), and
  RFC 3161 timestamps (backdating) remain the ADR-027 deferrals that would close
  these, each with its stated trigger.

## Alternatives Considered

1. A second serialization in the verify path (re-derive the bytes independently
   as a cross-check). Rejected: it reintroduces exactly the drift ADR-029
   removed; two serializations cannot be guaranteed equal, and a divergence
   reports a phantom break. One serializer, exercised by both paths, is the
   property.
2. verify_chain returning a bare boolean. Rejected: a boolean tells an auditor
   the chain broke but not where; the first-break report (index, sequence,
   expected vs found) is the diagnosable form the phase exists to provide.
3. A full walk at every store open. Rejected: it makes startup cost grow with
   the trail. The tail window bounds open-time cost; the full walk is the
   auditor's explicit CLI command.
4. verify-audit opening the store (reusing recovery and seeding). Rejected: an
   audit must not mutate the thing it audits; opening would trigger recovery and
   append a recovery event. verify_chain_file reads non-mutatingly.
5. A strict scalar-only payload allowlist. Rejected here because the EINGANG
   event carries a flat counts map and changing it is out of scope; a flat
   one-level container of simple leaves stays content-free by the same bound.
   Deeper nesting stays rejected.
6. Hashing the payload to a single fingerprint instead of restricting its
   shape. Rejected: a fingerprint loses the counts and versions that make the
   chain useful operationally, and the allowlist already keeps the shape
   content-free without discarding the data.
7. A per-event external anchor (anchor every append). Rejected for this phase as
   disproportionate: it is effectively the dedicated anchor command (backlog).
   The eval-time anchor demonstrates the concept at the cadence the repo already
   commits.
8. Writing startup_config into the chain for every command. Rejected: the
   read-only commands (show-document, verify-audit) have no chain to write into
   and must not open or mutate the audit store; the process command is where the
   chain lives.

## Consequences

Positive:

- A break is now diagnosable with location, at open (tail) and on demand (full
  CLI), using the one canonical serializer, so the proof has a single
  definition.
- The chain is content-free by construction, which is the precondition for it
  coexisting with the right to erasure.
- The git history is an external witness against truncation and rewrite, within
  its stated sparse, same-actor limits.
- The active controls are attested in the chain at process start, surviving
  beyond the retention-bound log.

Negative:

- The v1 payload allowlist is now a constraint on every event: a future event
  needing a richer payload must either fit the simple-types-plus-flat-container
  shape or change this policy. Intended.
- The tail window does not see a break before it at open; that is the
  startup-speed cut, covered by the full CLI walk.
- The anchor is sparse and same-actor in this repo; it does not close
  truncation, only raises its cost. Recorded, not overclaimed.

Neutral:

- STARTKONFIGURATION is a new AuditEventType, a process-wide custody event like
  WIEDERHERSTELLUNG. The system-id sentinel is generalized into core/events.py.
- The fail-closed raise in _emit, the chain-based read-access audit event, and a
  dedicated anchor command remain Round 19 / backlog.
