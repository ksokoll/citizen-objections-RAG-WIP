## ADR-002: Predefined Catalog as Matching Backbone

**Status:** Accepted

**Context:** Incoming objections follow recognizable patterns, particularly in coordinated campaigns where thousands of documents share near-identical arguments. The system must match incoming objections to known argument types to enable batch processing and response drafting.

**Decision:** The system uses a predefined, maintainable catalog of known objection patterns. Each catalog entry contains an argument type description, the relevant legal domain, and retrieval guidance. Incoming objections are matched against this catalog. Documents that produce no match generate a `NoMatchEvent` rather than a response draft.

**Rationale:** Predefined catalog entries produce auditable matching decisions with explicit confidence scores. Emergent clustering (dynamically discovering argument types without prior definition) cannot produce the same level of explainability: a cluster has no inherent label, no associated legal domain, and no defined response strategy. In a regulated context, the matching decision must be reconstructible months after the fact. A predefined catalog also makes the `NoMatchEvent` the primary signal for catalog extension, which is the most valuable feedback loop for system improvement over time.

**Rejected Alternatives:** Embedding-based dynamic clustering of incoming documents. Rejected because cluster labels are not interpretable without post-hoc human annotation, which undermines the audit requirement.

**Consequences:** A catalog management workflow is required: who adds entries, under what conditions, with what approval. Non-matching documents are not silently dropped but generate an explicit `NoMatchEvent` with the raw extracted arguments attached, to support Sachbearbeiter-driven catalog extension.
