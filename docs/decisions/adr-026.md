# ADR-026: Observability Logging Policy: One Sink, Default-Deny

Status: Accepted. The SRE parts (time-based rotation and the retention sweep,
the 0o600/0o700 file-permission hardening, the log_sink_size_bytes metric, and
the guarded fail-loud bootstrap) were superseded by the Round 21b rollback; see
the banner below. The governed-sink core stands.
Date: 2026-06-09
Deciders: Kevin Sokoll

> Round 21b rollback (2026-06-17). A second ownership pass removed the logging
> layer's SRE parts as out of demo scope and target-far for an ML/AI portfolio,
> the same scope discipline as the Round 21 chain rollback (ADR-030 superseded).
> What this ADR still describes correctly, and what it no longer does:
>
> - Removed: time-based rotation (TimedRotatingFileHandler) and the
>   sweep_expired_logs retention sweep. A single plain logging.FileHandler now
>   appends to one observability.log; there is no rotation and no time-based
>   retention. Sections "Retention" and "Single-process assumption" below are
>   superseded.
> - Removed: the owner-only file posture (0o700 directory, 0o600 files) and the
>   world-readable self-check. The sink is created with the default umask. The
>   PII masking, not OS file permissions, is what keeps personal data out of the
>   logs. Section "Sink file permissions" below is superseded.
> - Removed: the log_sink_size_bytes startup event. It monitored an infra
>   symptom (unbounded growth under a failed rotation) that no longer applies
>   once rotation is gone.
> - Removed: ObservabilityBootstrapError and the heavy guard that translated a
>   directory, handler, or structlog setup failure into a named, actionable
>   abort. configure_logging now sets up the sink directly; a setup failure
>   propagates the underlying error. The "Strict bootstrap" subsection is
>   trimmed accordingly below.
>
> Kept, because it is structure-thinking that transfers to ML systems rather
> than SRE trivia, and unchanged in behavior: the single governed sink, the
> default-deny key allowlist, the registered per-context event vocabulary,
> exception reduction to type plus location, value sanitization, the strict-mode
> vs production split, and never_raise. never_raise in particular is reframed
> here (see "Unbreakable runtime") as a systems-design decision, the
> instrumentation path must not tear down the business path, not infra
> robustness. The default-deny allowlist self-check stays as a bare configure-
> time check raising ProcessorChainError. fail-closed and
> audit_write_failures_total (the audit cluster) are untouched.
>
> Deferral record (built, then deliberately rolled back for demo scope;
> production would restore them):
> - operational retention: time-based rotation plus the mtime sweep.
> - file ACLs: the owner-only directory and file posture and the world-readable
>   self-check.
> - the sink-size startup metric (a rotation-failure signal).
> - a guarded fail-loud bootstrap: setup failures translated into a named,
>   actionable startup abort instead of a raw propagated error.
>
> This is a stronger deferral than "not built": it shows command of the topic
> plus the scope discipline to remove what the demo does not need.

## Context

The pipeline carries pseudonymous personal data (the document_id keys a raw
store that still holds PII until erasure; ADR-025, OBSERVABILITY_IMPLEMENTATION.md).
Structured logs are therefore a third store of pseudonymous data alongside the
audit chain and the raw store, and they are the one store written on every code
path, by our code and by third-party libraries alike. A logging layer that only
governs our own structlog calls is a leak waiting to happen: Presidio,
OpenTelemetry, and urllib3 log through the stdlib root logger, bypassing any
structlog-only allowlist, and an exception logged with str(e) or a rendered
traceback carries foreign-authored text of unknown content straight to disk.

The failure modes an external review surfaced, all of which this ADR closes:

- A structlog allowlist alone is bypassable by stdlib logging from third-party
  libraries.
- Event names and exception fields are uncontrolled free-text channels.
- Size-based rotation does not implement a time-bound retention obligation.
- The previous Pipeline._emit fallback printed str(e) to stderr, which both
  violated the exception policy and bypassed every control via a channel
  logging cannot reach.

## Decision

Default-deny, enforced at the single point all log output passes through. One
sink, one processor chain.

### One sink architecture

structlog routes into stdlib logging via
`ProcessorFormatter.wrap_for_formatter`; foreign stdlib records route through
the same shared processors via `ProcessorFormatter(foreign_pre_chain=shared)`.
A single plain `logging.FileHandler` on the root logger, appending to one log
file, is the only output (Round 21b: was a `TimedRotatingFileHandler`).
Because both worlds pass through one shared processor chain before one handler,
the controls below are enforced once and cannot be bypassed by choosing the
other logging API.

Third-party loggers are clamped to WARNING so their INFO/DEBUG chatter never
reaches the sink; what does reach it still passes the allowlist.

### Default-deny by origin and the lift-stamp-filter ordering rule

Default-deny applies to two separate questions, and they need two separate
controls: the key allowlist controls *what* may enter the record; the origin
rules control *who* may write a field. The shared chain is ordered to enforce
both, by the rule: lift untrusted data first or not at all, stamp truth after,
filter last.

- Lift first or not at all. Foreign-origin data carries no trusted diagnostic
  value and real forgery potential into governed fields, so it is not lifted at
  all: the chain has no extra-merging processor (no `ExtraAdder`), and a foreign
  record's `extra={...}` never reaches the event dict. This is a tightening of
  the original Round A design, where `ExtraAdder` lifted foreign extras and the
  allowlist dropped them: lifting then dropping let a foreign `extra` overwrite
  an authoritative field (`correlation_id`) or inject an allowlisted key before
  the allowlist ran. Not lifting closes that by origin.
- Stamp truth after. The authoritative fields (`correlation_id`, `level`,
  `timestamp`) are assigned unconditionally from their source of truth (the
  correlation ContextVar; the log method; the clock), overwriting any
  pre-existing value an own-code kwarg placed in the dict. `add_correlation_id`
  in particular assigns the ContextVar value when one is set and removes any
  pre-existing `correlation_id` otherwise, so an untrusted id can never survive.
- Filter last. Value normalization then the key allowlist run last, on a dict
  that already holds only trusted-origin data.

### Key allowlist (default-deny)

The shared chain's last processor before handoff keeps only an explicit
frozen set of keys and drops every other. A new field is invisible until it is
added to the allowlist on purpose. With foreign extras no longer lifted, the
allowlist is the backstop for own-code and structlog-internal keys rather than
the foreign-extra gate, but it remains load-bearing: a genuinely new field is
invisible until allowlisted on purpose. The allowlist is frozen by a golden
test so a key cannot be added without the test changing.

The drop is mode-dependent, the sibling of the message-policy raise (Round 17,
finding 2). In strict mode (CI) an own event carrying a key that is not
allowlisted raises UnregisteredLogKeyError, so a mistyped or unallowlisted
field fails at its origin instead of vanishing silently from the line, the same
failure shape the project closes at every other seam. In production the
non-allowlisted key is dropped, never raised, so a stray key cannot abort the
request path (unbreakable runtime). The raise is gated on own code via the
_from_structlog discriminant; a foreign record never raises. This is the
intermediate level: the central allowlist stays and the silent drop becomes
loud. A per-event key schema (each registered event declaring its own fields)
is a deliberate backlog item, a strict superset that adds nothing to rebuild,
deferred while one reviewer can keep the small own-event field set in view and
the fields are operational metadata rather than cross-event confidentiality
boundaries.

### Value normalization (sanitize_values)

The processor immediately before the allowlist normalizes every string value.
It strips the C0 control characters (`\x00`-`\x1f`, including newlines,
carriage returns, and tabs) from all string values, so a foreign record cannot
forge a second log line or inject terminal control sequences through the `event`
message or any other string field. It also caps a foreign record's `event`
value (the arbitrary third-party message text) at 200 characters with a literal
`[truncated]` marker, bounding the foreign-message residual below. Our own
events are registered constants, so the length cap targets foreign messages
only.

### Sink file permissions (superseded, Round 21b)

Originally the sink was held to the raw store's owner-only posture (ADR-025): on
POSIX the log directory was created mode 0o700 and every log file opened mode
0o600 by a handler subclass overriding `_open`, with a startup world-readable
self-check. Round 21b rolled this back: the sink is created with the default
umask and there is no self-check. The PII masking (ADR-025), not OS file
permissions, is what keeps personal data out of the logs, so OS-level file
hardening was out of demo scope. Production would restore the owner-only posture
(deferral record in the banner).

### Renderer

JSON is the default renderer and is mandatory in security-relevant
environments: it escapes values predictably and is the format the residual
controls above are reasoned against. The console renderer is a developer
convenience only and is not for security-relevant deployments.

Format selection is a CLI decision only (Round 16.1, S7), superseding the
`OBSERVABILITY_FORMAT` switch of ADR-023 (decision 1): the `--log-format`
flag, default json, console an explicit opt-in. The former
`OBSERVABILITY_FORMAT` environment fallback was removed, because an
environment variable could silently switch a security-relevant deployment to
the console renderer without any flag in the invocation showing it. The
active format is recorded in the startup_config event, so which renderer
produced a sink's lines is determinable afterward.

### Message policy

Event names are registered static constants (observability/events.py), never
interpolated free text. Variable data goes into named, allowlisted fields, not
into the message string. The shared chain rejects any structlog event whose
name is not in REGISTERED_EVENTS (UnregisteredLogEventError), so an ad hoc or
typo'd message fails loudly instead of silently widening the vocabulary.
Foreign records are exempt from vocabulary enforcement (their message is
arbitrary by nature); they are discriminated from structlog events by the
`_from_structlog` marker that ProcessorFormatter sets on foreign records only.

### Exception policy

Exceptions are logged as type plus location (`exc_type`, `exc_location`), never
`str(e)` and never a rendered traceback. Rationale: an exception message is
foreign-authored text. A ValueError raised deep in a library can contain
whatever string the library chose to interpolate, including a fragment of the
input under processing. Type plus the originating file and line is enough to
diagnose and carries no content. The shared chain's exception-reduction
processor extracts those two fields from exc_info and discards the rest before
the record is rendered.

### Retention (superseded, Round 21b)

Originally time-based rotation was the retention mechanism: a
`TimedRotatingFileHandler` rotated at midnight UTC and kept `RETENTION_DAYS`
backups, and a startup sweep (`sweep_expired_logs`) deleted rotated files whose
mtime was past the retention. Round 21b rolled this back: a single plain file
handler appends to one log file, with no rotation and no time-based retention.
A single log file is sufficient for the demo; operational retention is a
deferral (banner), restored in production where the storage-limitation
obligation for this third store of pseudonymous data is enforced by an actual
mechanism rather than left conceptual.

### Three-stores lifecycle

The architecture keeps three stores with deliberately different lifecycles:

- The audit chain is undeletable (append-only, hash-chained) and content-free
  of PII: it holds only the pseudonymous document_id and operational metadata.
- The logs are pseudonymous (document_id, no PII payloads under the allowlist);
  they are technically deletable, and operational retention (rotation plus
  sweep) is a Round 21b deferral restored in production.
- The raw store holds full content and is the only store with an erasure path;
  deleting a raw-store entry severs re-identifiability in the chain and the
  logs at once, because both carry only the now-dangling token.

This split is what lets append-only tamper-evidence and storage limitation
coexist (the defensibility backbone in OBSERVABILITY_IMPLEMENTATION.md).

## Phase separation: strict bootstrap, unbreakable runtime

The enforcement above is split into two phases with opposite failure postures,
because a control belongs where its error originates, not where it happens to
surface. Strict at configuration time, unbreakable at request time.

### Strict bootstrap, no degradation

Configuration is an explicit composition-root call. The CLI entrypoint
(python -m app) calls configure_logging(log_dir=...) before any pipeline
wiring or context code runs, so no log can escape the default-deny chain; the
sink path is a parameter resolved at the entrypoint, never an environment
read at import time (Round B; security finding 5, path injection via the
process environment). The behavior flags follow the same rule (Round 17,
finding 8): strict mode and the tracing flag are resolved once at the root,
where the CLI reads OBSERVABILITY_STRICT and OBSERVABILITY_TRACING and wires
them (configure_logging's strict parameter, set_tracing_enabled), and the chain
and the traced decorator read the wired flags thereafter, never live from
os.environ deep in the stack. A behavior switch is wiring the composition root
owns, not an ambient global any layer can reach. The Round 15.2 interim,
configuration as an import-time side effect of the logging module, is retired:
importing the observability package installs nothing. The test suite configures via an explicit session
fixture in conftest. Bootstrap fails loud: configure_logging sets up the sink
directly, and a directory, structlog, or handler setup failure propagates the
underlying error, while a missing default-deny allowlist raises
ProcessorChainError, which the CLI turns into a clean startup abort with a
nonzero exit. (Round 21b removed the ObservabilityBootstrapError translation:
the demo does not need a named, actionable abort for every setup failure; the
raw error propagates. The deferral record is in the banner.) The allowlist
self-check therefore runs once at configure time, not per event; post-startup
chain tampering is out of scope for the runtime path.

There is deliberately no degradation to a NullHandler or to bare stderr. The
logs are the one store written on every code path and the governed sink is the
control that keeps pseudonymous data inside the allowlist. Running the pipeline
without that sink would be fail-open for the central PII control, so a bootstrap
failure must stop the process, not silently downgrade it. The cost (an
unconfigurable environment cannot start) is the intended one: an operator fixes
the path or permissions rather than running blind.

### Unbreakable runtime

Once configured, no logging call may abort a business operation. Every own
processor is wrapped by never_raise: a processor exception is contained as a
substitute processor_failed event naming the failing processor, and the
business call returns. never_raise is a systems-design decision, not infra
robustness (Round 21b reframe): the instrumentation path must not tear down the
business path, the same failure mode an ML engineer meets directly when a
monitoring error kills a pipeline. The event-vocabulary check is mode-dependent: in strict
mode (the wired strict flag, set by the test conftest) an unregistered name
raises so CI catches every typo at its origin; in production the original name
is discarded entirely (it is potential payload, e.g. an interpolated f-string)
and replaced by the unregistered_log_event constant plus the caller location.
The key allowlist now follows the same mode-dependence (the unregistered-key
raise above), so name and key enforcement are symmetric.
Enforcement of the vocabulary thus lives in CI, where a typo is a fixable
defect, not in the request path, where it would convert a logging mistake in a
rarely exercised branch into a failed objection.

## Mechanisms, not conventions

Every control has a test that asserts its behavior at the sink, not at the call
site. The configure-time allowlist self-check, the never_raise containment, and
the production substitution each have a test, so a later logging refactor, a
test setup, or a migration script cannot silently remove a control without a
test changing.

## Residual channels (named, not hidden)

Default-deny at the sink does not make the logs unconditionally content-free.
The remaining channels, each with its mitigation:

- Foreign message text. A third-party WARNING/ERROR record's message string
  passes through as the `event` value. It is now bounded, not merely documented:
  the WARNING clamp drops INFO/DEBUG chatter, control characters are stripped,
  and the value is length-capped at 200 characters with a `[truncated]` marker.
  The bound is mitigation, not closure: the content is still not inspected or
  redacted, so this remains the weakest residual. Full closure stays with the
  deferred periodic log-sink scan and redaction (ADR-027); the bound reduces the
  blast radius until then.
- Values under allowlisted keys. An allowlisted field could in principle carry
  more than intended. Mitigated by keeping the allowlisted set operational
  (counts, ids, status), by type and length restriction at the call site, and
  by review.
- `print()` and C-level stderr. Output that bypasses the logging module
  entirely is out of logging's reach by construction. The mitigation is a
  no-`print()`-in-src grep check, not a runtime control.

## Alternatives Considered

1. structlog-only allowlist (no foreign routing). Rejected: leaves the stdlib
   bypass open, which is the main leak path given Presidio and OTel.
2. A separate scrubbing pass over the log file after the fact. Rejected as the
   primary control: it is reactive, races the reader, and a missed run leaks.
   Retained only as deferred defense-in-depth (ADR-027).
3. Size-based RotatingFileHandler. Rejected for retention: size says nothing
   about age, and the obligation is time-bound. Time-based rotation plus an
   mtime sweep expresses the obligation directly.
4. Rendering tracebacks for debuggability. Rejected: a traceback is the richest
   foreign-text channel of all. Type plus location keeps diagnosability without
   the content.

## Consequences

Positive:
- Both logging worlds are governed by one allowlist; there is no second API
  that escapes it.
- PII discipline, event vocabulary, and exception reduction are mechanisms with
  sink-level tests, not documented conventions.
- Operational retention was built as an actual mechanism (rotation plus sweep),
  then rolled back for demo scope (Round 21b); production restores it (banner).

Negative:
- Foreign message text is a named residual that the allowlist cannot inspect;
  it is bounded (WARNING clamp, control-character strip, length cap) but closing
  it fully needs the deferred sink scan and redaction.
- The allowlist is a maintenance point: a genuinely new operational field
  requires a deliberate allowlist change and a golden-test update. This is the
  intended cost of default-deny.

Neutral:
- The vocabulary and allowlist couple the logging layer to a small registry
  that must grow deliberately as new governed events are added in Rounds B
  and C.
- The import-time side effect assumes a single synchronous entry point; an
  async or multi-process entry point would require revisiting the contextvar
  propagation and the single-handler assumption (recorded in the round-15
  pre-registration Open Risks).

## Single-process assumption: no second handle on the active file (superseded, Round 21b)

This section addressed a rotation failure mode: TimedRotatingFileHandler rotates
by renaming the active file at midnight, and on Windows that rename fails if a
second process holds the file open (PermissionError, no rotated file, the active
file kept growing), so the retention obligation silently failed on that
platform. The assumption was made observable by the log_sink_size_bytes startup
event. Round 21b removed rotation and the size event, so the failure mode no
longer exists: a plain file handler appending to one file performs no rename and
has nothing for a second handle to block. The concern returns only with the
deferred operational retention (banner), at which point this assumption and a
Windows rotation fallback would be revisited.

## Declared assumption: single synchronous run

Two pieces of the instrumentation are deliberately global, and that is correct
only while one run executes at a time. The in-memory span exporter is module
held and cleared by the Coordinator at run start (clear_finished_spans), and
the metric registry is a single in-process CollectorRegistry. Under the demo's
single synchronous pass both are right: the finished spans belong to the one
run in flight, and counters are cumulative across the fleet by design, so a
process-wide registry is what "count the fleet" means.

This is a declared simplification, not silent debt (Round 17, finding 1).
Building a run-scoped span collector and injecting a per-composition registry
now would be mechanism on spec against a scenario that does not occur, so it is
not built (YAGNI); the assumption is recorded here in the same move so the debt
stays visible rather than being discovered later. The assumption breaks the
moment runs overlap: a concurrent entry point (a FastAPI request handler, a
worker pool) or a metrics scrape endpoint would let one run's
clear_finished_spans discard another run's spans, and would share the global
registry across requests in ways the single-run code does not anticipate. At
that point the retrofit is named: a run-scoped span collector that filters the
exporter by the run's trace id instead of clearing it globally, and registry
injection at the composition root instead of a module global. The correlation
id needs no such retrofit: it is already a ContextVar, ambient per execution
context and therefore concurrency safe as is.

The audit chain shares this single-synchronous-run assumption and now states it
without a guard. ADR-030's single-writer advisory lock was rolled back in Round
21 as out of demo scope (ADR-030 superseded): the demo has one synchronous
writer, so the lock was functionless, and the chain is therefore not guarded
against a concurrent writer. The in-memory head is the sole duplicate mechanism,
so two writers on one path would interleave appends and leave the head stale.
The trigger that would restore the lock is the same concurrent entry point named
above (a request handler, a worker pool); until then the assumption is recorded
rather than enforced.

## Backlog with named triggers (Round 17.1)

Round 17.1 moved domain vocabulary into the contexts (event constants and
allowlisted keys per context, unioned at the composition root) and made the
telemetry imports lazy, but it deliberately did not close two further items.
Each is recorded here as a declared simplification with a named trigger, not
silent debt:

- **Run-scoped instrumentation object (concurrency retrofit).** Round 17.1
  added two more process-wide observability globals: the event registry and the
  key allowlist, both mutable module sets assembled at the root. Like the span
  exporter and the metric registry above, they are correct under the single
  synchronous run and break the moment runs overlap with different vocabularies.
  The retrofit is the same named one: a run-scoped instrumentation object that
  holds the registries (and the span collector and metric registry) per
  composition instead of as module globals, injected at the root. Trigger: a
  concurrent entry point (a FastAPI handler, a worker pool) or a metrics scrape
  endpoint. Until then the symmetric autouse test reset
  (conftest._reset_observability_globals) keeps the test suite order
  independent, which is the part of the debt that pays today.
- **logging_config.py responsibility split.** logging_config.py carries the
  processors, the sink/handler, and configuration in one module (Round 21b
  removed retention and the heavy bootstrap guard, so the module is smaller now).
  Round 17.1 changed where the allowlist comes from (a root-assembled union),
  not the module's internal layout, to keep the diff reviewable. The split into
  responsibility modules (processors, sink/handler, configuration) is backlog.
  Trigger: when the module is next substantially edited.
