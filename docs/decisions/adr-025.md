# ADR-025: PII Masking via Layered Detection (Presidio plus Anchor Extraction)

Status: Accepted
Date: 2026-06-03
Deciders: Kevin Sokoll

## Context

ADR-010 established that DocumentIngestion masks PII as the last step before
handing text downstream, that masking (reversible via an access-controlled
raw store) is chosen over irreversible anonymization, and that masking
accuracy is part of the Architecture Fitness Functions. Until now the step
was a pass-through (clean_text == raw_text); the raw Einwendung text flowed
through Triage, Retrieval, Briefing, and the AuditLog unmasked.

The task: detect and mask PII in free-text German Einwendungen,
deterministically and without an LLM in the masking step itself.

A note on the threat model, which changed during this work and relaxed the
requirements. The production deployment assumes the Triage LLM (Mistral) runs
in an encapsulated environment (no outbound network, no prompt retention,
on-prem or VPC). Under that assumption the masking does not protect against a
third-party processor; it is internal data minimization and least privilege
(limit PII in logs and to case workers, reduce blast radius if the boundary
breaks). A missed entity stays within the trust boundary. The data-protection
reasoning is in DATA_GOVERNANCE_STATEMENT.md. Named-entity masking is
de-identification, not anonymization; residual re-identification risk from
quasi-identifiers and LLM inference is documented there, not solved by more
masking.

## Decision

Layered detection behind a PiiMasker Protocol, with the masker as the only
module importing Presidio.

Masked scope (identifying core attributes only): PERSON (name), PHONE_NUMBER,
EMAIL_ADDRESS, IBAN_CODE. Locations, postal codes, and case numbers are
deliberately NOT masked (see Rationale).

Detection layers:
1. Anchor-based zone extraction (zone_extractor, pure regex, no spaCy). Names
   in the structurally fixed zones (submitter line "Einreicher:/Von:/
   Einreichende Person:", representative clause "vertreten durch ...") are
   extracted deterministically. A shared prefix set strips titles and
   functions (Dr., Dipl.-Ing., Geschäftsführer, den Vorsitzenden, ...). An
   organisation before "vertreten durch" is skipped; the natural person after
   it is taken. Extracted names are masked at every word-boundary occurrence,
   which also covers the signature without a separate anchor.
2. Presidio analyze over the full text: spaCy German NER (de_core_news_md) for
   PERSON, a controlled German phone-number regex, and the built-in email and
   IBAN recognizers. The built-in PhoneRecognizer is removed from the registry
   because it matches German date formats (e.g. "08.11.2024") as phone
   numbers; only the controlled regex decides what is a phone number.

The two sources are merged additively (anchor spans plus analyzer spans), and
a single Anonymizer pass replaces detected spans with speaking German
placeholders ([NAME], [TELEFON], [EMAIL], [IBAN]), not empty strings.

Count contract (owned by our code, not by Presidio): one count per masked
region after our own resolution. The masker merges the anchor and analyzer
spans into regions (joining overlapping spans, and whitespace-adjacent spans of
the same type), counts the regions per type, and hands that same merged set to
the anonymizer. Merging the union (not dropping the lower-scored span) keeps the
masked extent intact, so the masked text and the recall are unchanged; the
whitespace bridge makes a multi-token name one region regardless of whether the
NER spanned both tokens, so the count is deterministic and not an artefact of
model behaviour. The audit value masked_entity_counts is DSGVO-relevant, so it
must not rest on a third-party heuristic (the anonymizer's internal merge of
anonymized.items) that a Presidio version bump could shift silently. The same
name in both the submitter header and the signature is two regions, hence two
NAME counts; a medium-scale test against the real PresidioMasker pins this. The
counts are identical to those the previous anonymized.items derivation produced
on the evaluation corpus: this change relocates and pins the contract, it does
not move the numbers.

Coverage self-check and audit evidence. The anchor layer is deterministic:
every non-stopword token of every extracted name is scheduled for masking. The
masker verifies this held in the output. After anonymization it checks, with
the same word-boundary logic the recall evaluation uses (_is_present_as_word),
that no anchor name token survives as a word; a survivor is an internal
contradiction (an offset, overlap-resolution, or anonymizer bug), so it logs a
stderr warning naming the masked-name and survivor counts. It never raises and
never blocks: under the encapsulated-LLM model a slipped name stays inside the
trust boundary, so a leak must be provable, not fatal. The positive evidence is
the NAME entry in masked_entity_counts, already carried into the EINGANG audit
event; the count is now read as deterministic-coverage evidence, not only as a
Fitness-Function input. The self-check is the anchor layer's own contract; it
does not detect names that exist only in running text and are missed by spaCy
(that residual stays in Consequences and DATA_GOVERNANCE_STATEMENT). The helper
is duplicated from the experiment evaluator rather than imported, because
production code must not depend on the offline experiment package.

Masking is one-way: no token-to-original mapping is kept. The original is
available only from the raw store via the document_id, for archival and
Sachbearbeiter audit, never re-injected into the pipeline. The raw store is
created owner-restricted on POSIX (directory 0o700, files 0o600) and a startup
check warns if it is world-readable; on Windows POSIX modes do not map to ACLs,
so the enforcement is best-effort and the access claim is only as strong as the
platform supports.

Structure:
- PiiMasker Protocol (protocols.py) and MaskingResult DTO (entities.py) live in
  the DocumentIngestion context. They never cross a context boundary: the
  Protocol is consumed only by DocumentIngestionService and implemented only by
  PresidioMasker and FakePiiMasker, and MaskingResult is read into the
  cross-context IngestionResult inside the service. Only IngestionResult stays
  in core, where the Coordinator reads it. The pre-existing Retriever protocol
  has the same core-placement tension and deserves the same treatment later.
- PresidioMasker and zone_extractor in the DocumentIngestion context.
- DocumentIngestionService takes a PiiMasker by injection; it stores the raw
  text first, then masks, so the original is in the store before any masking.

Dependencies are pinned (Weg B, versioned for deterministic NER): the spaCy
model is a PEP 508 direct reference in pyproject.toml
(de_core_news_md @ <github release wheel url>), not a loose
"python -m spacy download". presidio-analyzer and presidio-anonymizer are
pinned exactly (==), like the model wheel, because masked_entity_counts depends
on their behaviour. Pinning makes the NER output and the count contract
deterministic, which is what makes the golden evaluation stable.

## Rationale

- Why layered, not flat NER. Measured: flat NER reaches only about 75% name
  recall on the evaluation corpus (de_core_news_md and de_core_news_lg both,
  so model size is not the lever). The misses concentrate in the structurally
  fixed submitter and representative zones, where a name right after a title
  in a header line is out-of-distribution for a model trained on running text.
  NER is the wrong primary tool for a structured zone. Anchor extraction makes
  those zones a near-deterministic operation; NER then handles only the
  running text, where it is in its element. The layered masker reaches 92.3%
  name recall at 98.0% precision on the same corpus (see Consequences). This
  is the "anchor and propagate" pattern applied to person zones.
- Why the reduced scope. LOCATION masking was empirically catastrophic for
  utility: against a real document, blanket location masking produced 26
  masked spans, destroying argument-bearing geography (river names, protected
  areas, plan areas) that the Triage needs. A place name barely identifies in
  a regional mass objection, and locations are the substantive subject of the
  objection. Postal codes alone do not identify once the name is masked.
  Case numbers (Aktenzeichen) are rare in raw citizen free text, are process
  identifiers rather than PII, and carry the highest false-positive risk
  against norm citations. All three were removed: delete, not refine.
- Speaking placeholders over empty strings: an empty string destroys sentence
  structure, giving the Triage LLM malformed input. A placeholder preserves
  syntax and is auditable (the masked document shows what type was removed).
- One-way masking: ADR-010 keeps the original in the raw store, so the
  pipeline text never needs de-masking; the masker keeps no sensitive mapping.

## Alternatives Considered

1. Layered detection (chosen). Deterministic anchors for the fixed zones plus
   Presidio NER/regex for the rest.
2. Flat NER (Presidio over the whole text, no anchors). Measured at about 75%
   recall; misses the fixed-zone names. Rejected as the primary mechanism.
3. Larger spaCy model (lg) instead of layering. Measured: no meaningful gain
   over md (both about 75%), only shifts which names leak. Rejected; lg is
   567 MB for no benefit.
4. Transformer NER (e.g. de_dep_news_trf or a HuggingFace model). Would raise
   recall on the running-text part, at higher cost and size. Noted as the
   production upgrade path, not built for the demo.
5. LLM-based masking. Highest recall, but violates the no-LLM constraint for
   the masking step, sends PII to a model, adds cost and variance.
6. No Protocol, call Presidio directly in the service. Rejected: violates the
   External I/O Boundary rule (unit tests could not run without spaCy). See
   the conflict resolution below.

## Conflict: Protocol vs. Simplify Loop

The Simplify Loop warns against abstractions that serve only one concrete
implementation. The External I/O Boundary rule requires external I/O behind a
Protocol so unit tests run without heavy models. These conflict for PiiMasker,
which has one production implementation.

Resolution: the External I/O rule wins, because its benefit is concrete. The
Protocol has two real users from day one: PresidioMasker in production and
FakePiiMasker in the service unit tests. The Fake is the second
implementation, so the Protocol is not over-engineered.

## Consequences

Positive:
- The pipeline is PII-free downstream of ingestion for the masked scope;
  Triage, Retrieval, Briefing, and AuditLog see only masked text.
- The service is unit-testable with a FakePiiMasker, no spaCy in fast tests.
- Measured masking quality (Fitness Function, experiments/pii_evaluation):
  92.3% name recall, 98.0% precision on a 20-document synthetic corpus. The
  measurement is deterministic (pinned spaCy model), so a single run per
  configuration is valid; no variance bound is needed.

Negative:
- Residual recall gap (about 8%): names in unanchored zones (a 12-line
  signature list, a "gez." lawyer signature) that NER misses. Acceptable under
  the least-privilege framing: a missed name stays within the trust boundary.
- Token-level masking of extracted names over-masks organisation names that
  share a token with a person name ("Weingut Groß", "Kanzlei Franken & Stein").
  Accepted as harmless over-masking (an org-name token masked, no leak, no
  argument damage).
- Dependencies: presidio-analyzer, presidio-anonymizer, and the pinned
  de_core_news_md wheel. The pinned model and the spaCy version must be kept
  in sync on upgrade.

Neutral:
- The Fitness Function reports recall and precision separately (not a single
  averaged score), because they capture opposite failure modes: recall is the
  safety axis (a leaked name), precision the utility axis (destroyed context).
  The corpus is synthetic (no real citizen PII), safe to commit and annotate.

Deliberately deferred (decisions, not omissions):
- No PersonSpanDetector strategy protocol. There is one masking backend, and a
  swap seam already exists at the PiiMasker Protocol. A second strategy
  abstraction inside the masker would serve one implementation, which the
  Simplify Loop warns against. Introduce it when a second detector (e.g. a
  transformer NER, alternative 4) is actually built.
- No externalised recognizer-vocabulary config. The anchors, title prefixes,
  and organisation markers stay as curated module constants. There is one
  corpus; externalising the vocabulary to config buys configurability nobody
  needs yet. Externalise when a second data source with different zone
  conventions arrives.
